# 프로젝트별 개발 프롬프트

3개 배치 · **개발 프로젝트 9개** (+ 선행 입력물 `my_cafe.md`).
각 프로젝트는 **① 스펙 생성 → ② 루프 실행** 2단계. 모든 가이드 파일은 `docs/guide/<slug>.md` 에 이미 들어 있다.

| # | 배치 | slug | 한 줄 목표 |
|---|---|---|---|
| 0 | 선행 | `my-cafe-concept` | AI 컨설턴트와 대화해 `my_cafe.md` 작성 (배치 2·3 입력) |
| 1 | 1 웹앱 | `gyebu-app` | 수입/지출 등록·조회·카테고리 합계 (Supabase) |
| 2 | 1 웹앱 | `community-app` | 로그인 + 게시글 CRUD(본인 글만 수정/삭제) |
| 3 | 1 웹앱 | `shopping-mall` | 상품·장바구니·합계 (결제 제외) |
| 4 | 1 웹앱 | `cafe-dashboard` | Auth+MCP+DB+AI 사장님 대시보드 (보스) |
| 5 | 2 에이전트 | `gyebu-agent` | 가계부 DB에 연결된 소비 분석 에이전트 |
| 6 | 2 에이전트 | `my-cafe-agent` | my_cafe.md + 운영 DB 결합 맞춤 에이전트 |
| 7 | 3 자동화 | `research-skill` | 웹리서치→md→노션 절차를 SKILL.md로 박제 |
| 8 | 3 문서 | `review-report` | 리뷰 CSV→엑셀 / 경쟁사 MD→PPT 보고서 |
| 9 | 3 자동화 | `instagram-influencer` | 인스타 인플루언서 발굴 Top5 + 컨택 초안 |

> 진행 권장 순서: **1 → 2 → 3** 검증 후 **4(보스)**, 그다음 **0 → 5·6**, 마지막 **7·8·9**.
> 공통 준비: `cp .env.sample .env && cp .env.sample .env.local` 후 값 채우기, MCP 설치(`스킬_MCP_설치안내.md`).

## 0. my-cafe-concept (선행, 루프 아님)

대화형으로 직접 만든다. `claude` 세션에 붙여넣기:
```
너는 카페 창업 컨설턴트야. 내가 열고 싶은 카페 컨셉을 한 번에 하나씩 질문해서 구체화해줘.
순서: ① 타겟 손님 → ② 위치·상권 → ③ 시그니처 메뉴 → ④ 분위기·인테리어 → ⑤ 가격대 → ⑥ 차별점 → ⑦ 이름·슬로건.
내가 막히면 보기를 2~3개 제안해줘. 다 끝나면 my_cafe.md 로 저장해줘
(형식: 카페이름/한 줄 컨셉/타겟/위치/시그니처/분위기/가격대/차별점/슬로건).
```
→ 산출물 `my_cafe.md` 를 배치 2·3 프로젝트 루트에 둔다.

## 1. gyebu-app  ⭐ 먼저 (스펙 예시 제공됨)

## 2. community-app

**① 스펙 생성** — `claude` 세션:
```
@docs/guide/community-app.md 를 읽고 @specs/gyebu-app.spec.md 와 동일한 형식으로
specs/community-app.spec.md 를 만들어줘. 코드는 만들지 말고 스펙만.
다음을 검증가능한 인수기준(AC)으로 변환하고 각 AC에 verifier 검증법을 붙여줘:
- 회원가입/로그인(Supabase Auth)
- 게시글 작성=로그인 사용자만, 조회=로그인 누구나(작성자 표시), 수정/삭제=본인 글만
- 게시글 목록 최신순(제목/작성자/시간), 클릭 시 상세
- Vercel 배포 URL 존재
스택: Next.js + Supabase. 범위 밖: 결제.
```
**② 루프 실행**
```bash
bash harness/run-loop.sh community-app 5
```

## 3. shopping-mall

```
@docs/guide/shopping-mall.md 를 읽고 @specs/gyebu-app.spec.md 형식으로
specs/shopping-mall.spec.md 를 만들어줘(코드 금지, 스펙만).

- Vercel 배포. 범위 밖: 결제(주문하기 버튼은 '준비 중' 처리).
스택: Next.js + Supabase.
```
**② 루프 실행**
```bash
bash harness/run-loop.sh shopping-mall 5
```

## 5. gyebu-agent  (배치 2 — QA는 질문 루브릭)

> 빌드 검증이 아니라 **샘플 질문→답변 품질**로 판정. 전제: gyebu-app 의 Supabase DB에 데이터 10건+.

**① 스펙 생성**:
```
@docs/guide/gyebu-agent.md 를 읽고 specs/gyebu-agent.spec.md 를 만들어줘(코드 금지).
이 프로젝트는 빌드가 아니라 'DB 연결 + 질문 응답 품질'이 핵심이다.
AC로 변환하고, verifier가 던질 '샘플 질문 세트'와 통과 루브릭을 포함해줘:
- 가계부 Supabase DB(같은 프로젝트/테이블)에 에이전트가 접속 (Supabase MCP)
- 기본 조회 질문 3개("이번 달 얼마?", "식비 최대일?", "교통비 월평균?") → DB 근거로 정확 답변
- 패턴 분석 2개(주중/주말, 카테고리 비율) → 데이터 기반
- 절약 조언 1개 → 데이터 근거 포함
루브릭: 각 답변이 실제 DB 수치와 일치 + 환각 없음 = PASS.
```
**② 실행** (대화형 conductor 권장):
```
conductor 에이전트로 gyebu-agent 진행. @specs/gyebu-agent.spec.md 의 질문셋을
verifier가 실제 DB 조회 결과와 대조해 VERDICT 판정하게 해줘.
```
또는 `bash harness/run-loop.sh gyebu-agent 5`

## 6. my-cafe-agent  (배치 2)

**① 스펙 생성**:
```
@docs/guide/my-cafe-agent.md 를 읽고 specs/my-cafe-agent.spec.md 를 만들어줘(코드 금지).
핵심: my_cafe.md(컨텍스트) + 운영 DB 결합 → 맞춤 답변. Before/After 비교.
AC + verifier 검증법:
- my_cafe.md 를 컨텍스트로 로드, 운영 DB(매출·메뉴·리뷰) 연결(Supabase MCP)
- 동일 질문을 'Context 없이' vs '있이' 비교 시연(최소 2쌍) → 있이가 더 맞춤형
- 맞춤 답변이 my_cafe.md의 타겟/시그니처/차별점을 반영
루브릭: Before/After 차이가 명확 + 답변이 내 카페 데이터 근거 = PASS.
```
**② 실행**: `bash harness/run-loop.sh my-cafe-agent 5` (또는 conductor 대화형)

## 7. research-skill  (배치 3 — 스킬 박제)

> 산출물이 코드가 아니라 `SKILL.md` + 재호출 동일성. 대화형이 더 적합.

**① 스펙 생성**:
```
@docs/guide/research-skill.md 를 읽고 specs/research-skill.spec.md 를 만들어줘(코드 금지).
AC + verifier 검증법:
- .claude/skills/research/SKILL.md 생성(name/description/절차 4단계: 주제질문→브라우저 MCP 탐색3곳→정해진 포맷 md→노션 공유)
- 새 대화에서 "리서치 돌려줘" 재호출 시 스킬 자동 로드 + 같은 포맷 결과
- 키워드만 바꿔 2번째 결과도 동일 포맷
루브릭: 재호출 2회 결과가 동일 포맷 = PASS.
전제: 브라우저 MCP 연결.
```
**② 실행** (대화형 권장):
```
conductor 에이전트로 research-skill 진행. builder가 SKILL.md를 만들고
verifier가 새 세션에서 재호출해 포맷 동일성을 확인하게 해줘. @specs/research-skill.spec.md 기준.
```

## 9. instagram-influencer  (배치 3 — 자동화)

> ⚠️ 본인 계정 직접 로그인, 자동 대량 스크래핑·DM 금지. 자동화는 후보 발굴·열람까지만.

**① 스펙 생성**:
```
@docs/guide/instagram-influencer.md 를 읽고 specs/instagram-influencer.spec.md 를 만들어줘(코드 금지).
AC + verifier 검증법:
- Playwright MCP로 인스타 후보 발굴(해시태그/경쟁사 태그)
- 후보별 팔로워/평균 좋아요·댓글/인게이지먼트율/적합도 정리
- influencers.md 에 Top5 표 + 상위3 DM 초안
- (가점) .xlsx/.pptx 또는 노션 공유
루브릭: Top5 표(지표 포함) + 컨택 초안 3개 생성 = PASS. 안전수칙 준수.
```
**② 실행** (대화형 권장 — 사람이 로그인 개입 필요):
```
conductor 에이전트로 instagram-influencer 진행. 내가 직접 인스타 로그인할 거고,
이후 후보 발굴·지표 계산·influencers.md 작성을 builder가, 산출물 검증을 verifier가 하게 해줘.
@specs/instagram-influencer.spec.md 기준, 자동 DM 금지.
```

### 팁
- 배치 1(웹앱)은 헤드리스 하네스가 잘 맞고, 배치 2·3은 사람 개입/판단 비중이 커서 **대화형 conductor**가 더 매끄럽다.
- 각 `specs/<slug>.spec.md` 가 생성되면 한 번 열어보고 AC가 통과/실패로 명확히 갈리는지 확인 후 루프를 돌려라.
