# 실행 프롬프트 모음 (Claude Code 터미널)

복붙해서 쓰는 프롬프트. 모든 프롬프트는 **가이드/스펙을 반드시 따르도록** 강제 문구를 포함한다.

---

## A. 가장 추천 — 하네스로 자동 루프 (강제·반복)
스펙/가이드를 매 단계 주입하고 develop→QA→판정을 5회까지 자동 반복.
```bash
# 1) 준비
cp .env.sample .env            # 값 채우기 (Supabase 등)
cp .env.sample .env.local      # 앱용
# specs/<project>.spec.md, docs/guide/<project>.md 배치 확인

# 2) 실행 (예: 가계부 앱)
bash harness/run-loop.sh gyebu-app 5

# 로그: harness/logs/gyebu-app/iterN-build.log , iterN-qa.log
# 성공 시 exit 0 / 5회 초과 실패 시 exit 2 (에스컬레이션)
```

---

## B. 대화형 — conductor에게 오케스트레이션 위임
`claude` 세션을 열고:
```
conductor 에이전트를 사용해 gyebu-app 프로젝트를 진행해줘.
반드시 @specs/gyebu-app.spec.md 와 @docs/guide/gyebu-app.md 만을 기준으로,
builder→verifier 루프를 돌려서 모든 AC가 VERDICT: PASS 가 될 때까지(최대 5회) 진행해.
스펙에 없는 기능은 추가하지 말고, 막히면 멈추고 무엇이 필요한지 보고해.
```

---

## C. 단계별 수동 호출 (디버깅용)
```
# 개발
builder 에이전트로 @specs/gyebu-app.spec.md 의 AC2(등록), AC3(조회)를 구현하고 npm run build 통과시켜줘.

# QA/판정
verifier 에이전트로 @specs/gyebu-app.spec.md 의 모든 AC를 실제 실행으로 검증하고,
AC별 PASS/FAIL 표 + REASONS + 마지막 줄 VERDICT: PASS|FAIL 로 보고해줘.
```

---

## D. 새 프로젝트 스펙 생성 (배치 1 나머지)
```
@docs/guide/community-app.md 를 읽고, gyebu-app.spec.md 와 동일한 형식으로
specs/community-app.spec.md 를 만들어줘. 미션/핵심구조/제출물을 모두
'검증 가능한 인수기준(AC)'으로 변환하고, 각 AC에 verifier 검증 방법을 붙여줘.
스펙 작성만 하고 코드는 만들지 마.
```
이후 `bash harness/run-loop.sh community-app 5`.

---

## 진행 순서 권장
1. `gyebu-app` 로 루프가 잘 도는지 검증(가장 단순).
2. 통과하면 `community-app` → `shopping-mall` 순서로 스펙 생성 후 루프.
3. 마지막에 `cafe-dashboard`(보스): 스펙이 크므로 verifier와 별도로 judge 에이전트 분리 고려.
