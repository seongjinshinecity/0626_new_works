# 루프 개발 시스템 — 개발 계획

AI 공장장 부트캠프 3기 퀘스트(가계부 앱 ~ 카페 사장님 대시보드)를 **3개 에이전트 루프**(지시→개발→QA/완료판단)로 자동 개발하기 위한 설계 문서.

> 핵심 원칙: **에이전트 수보다 "완료의 정의(spec)"가 중요하다.** 모든 단계는 폴더 안의 가이드/스펙을 **반드시** 따르도록 하네스가 강제한다.

---

## 1. 프로젝트 배치 그룹화

10개 콘텐츠 페이지를 스택 유사성으로 3개 배치로 묶고, 루프가 가장 잘 맞는 **배치 1부터** 시작한다.

### 배치 1 — 풀스택 웹앱 (Auth · DB · CRUD · 배포) ⭐ 먼저 진행
| 프로젝트 | 스택 | 비고 |
| --- | --- | --- |
| [Server+DB] 가계부 앱 | Next.js + Supabase | 루프 검증용 첫 타깃 (가장 단순) |
| [Auth] 커뮤니티 앱 | Next.js + Supabase Auth | 익명게시판 + 로그인/소유권 |
| [Auth+DB] 쇼핑몰(결제제외) | Next.js + Supabase | 상품/장바구니/합계 |
| [Auth+MCP+DB+App] 카페 대시보드 | Next.js + Supabase + MCP + AI | 보스 퀘스트, 마지막 |

공통: Next.js(React) + Supabase(DB/Auth) + Vercel 배포. **완료 기준이 객관적으로 검증 가능**(빌드 성공, CRUD 동작, DB 행 존재, 배포 URL)해서 루프에 최적.

### 배치 2 — 데이터 분석 에이전트 (Agent · DB · Context)
| 프로젝트 | 스택 |
| --- | --- |
| [Agent+DB] 가계부 분석 에이전트 | Supabase MCP + 분석 프롬프트 |
| [Context+Agent+DB] 내 카페를 아는 AI 에이전트 | Supabase MCP + `my_cafe.md` 컨텍스트 |

QA = 샘플 질문 세트를 던져 답변이 **데이터 근거 + 맞춤성** 루브릭을 충족하는지 판단.

### 배치 3 — 자동화 · 문서 산출 (Browser/Skill · Documents)
| 프로젝트 | 스택 |
| --- | --- |
| [My Agent] 자동 리서치 스킬 | Browser MCP + SKILL.md |
| [조사] 리뷰/경쟁사 → 엑셀·PPT | document skills(xlsx/pptx) |
| [홍보] 인스타 인플루언서 찾기 | Playwright MCP |

QA = 산출물 파일이 스펙대로 생성/실행되는지, 스킬 재호출 시 동일 포맷인지.

### 선행 입력물 (개발 대상 아님)
- **내 카페 컨셉 `my_cafe.md`** — 배치 2·3의 컨텍스트 입력. 루프 시작 전에 먼저 작성해 둔다.

## 2. 3개 에이전트 역할

| 에이전트 | 파일 | 역할 | 도구 권한 |
| --- | --- | --- | --- |
| **conductor** (지시/오케스트레이터) | `.claude/agents/conductor.md` | 스펙을 읽고 작업 분해, builder·verifier 호출, 판정 결과로 재지시, 반복/에스컬레이션 | Read/Grep/Bash + 서브에이전트 호출 |
| **builder** (개발) | `.claude/agents/builder.md` | 스펙의 미충족 항목을 구현/수정, 빌드 통과까지 | Read/Write/Edit/Bash/Grep |
| **verifier** (QA + 완료판단) | `.claude/agents/verifier.md` | 각 완료기준을 **실제 실행**으로 검증, 기계판독 `VERDICT: PASS/FAIL` 출력 | Read/Bash/Grep (Write 없음 → QA 정직성 확보) |

> verifier에 **Write/Edit 권한을 주지 않는 것**이 핵심. QA가 코드를 직접 고치면 "통과시키려 자가수정"하는 부정이 생긴다. 검증과 수정의 분리를 권한으로 강제한다.

초기엔 verifier가 완료판단을 겸한다. 카페 대시보드(보스)에서만 별도 judge 에이전트로 분리하는 것을 권장(스펙이 복잡해 판정 부하가 큼).

## 3. 루프 아키텍처 (하네스가 강제)

```
              ┌──────────── harness/run-loop.sh (반복·강제) ─────────────┐
              │  매 반복마다 @SPEC(가이드)을 프롬프트에 강제 주입          │
              ▼                                                          │
  [SPEC 로드] → builder: 미충족 항목 1개 구현 → verifier: 전 항목 실제검증 │
                                                  │                       │
                              VERDICT: PASS ───────┴─→ 종료 (성공)        │
                              VERDICT: FAIL → REASONS를 다음 builder에 피드백
                              반복 ≤ MAX_ITER(기본 5), 초과 시 사람에 에스컬레이션
              └──────────────────────────────────────────────────────────┘
```

**가드레일**
- `MAX_ITER`(기본 5) 초과 시 중단 + 실패 리포트 → 사람 개입
- verifier는 "괜찮아 보임"이 아니라 **실제 명령 실행**(빌드/기동/쿼리/CRUD 호출)으로만 판정
- verifier 출력 마지막 줄은 반드시 `VERDICT: PASS` 또는 `VERDICT: FAIL` (하네스가 grep으로 파싱)
- 모든 단계에 `@docs/spec/<project>.spec.md` + `@docs/guide/<project>.md`를 주입해 **가이드 이탈 방지**

## 4. 스펙(완료의 정의) 작성 규칙

각 프로젝트는 `specs/<project>.spec.md`에 **검증 가능한 인수기준(Acceptance Criteria)**으로 정의한다. 커리큘럼 가이드의 *미션 / 핵심 구조 / 제출물*을 그대로 기준으로 변환한다.

형식(각 항목은 통과/실패가 1:1로 판정 가능해야 함):
```
- [ ] AC1 (빌드): `npm run build` 성공 + 개발서버 기동
- [ ] AC2 (기능): <Given/When/Then> + verifier 검증 방법
- [ ] AC3 (DB): Supabase에 실제 행이 저장됨 (쿼리로 확인)
...
- [ ] ACn (제출물): README + 동작 스크린샷 경로
```

worked example: `specs/gyebu-app.spec.md`(가계부 앱) 참고. 나머지 배치 1 프로젝트는 같은 템플릿으로 작성.

## 5. 실행 순서 (요약)

1. `.env.sample` → `.env`로 복사 후 값 채우기 (Supabase 등)
2. MCP/스킬 설치 (`스킬_MCP_설치안내.md`)
3. `.claude/agents/*.md` 3종을 **개발할 프로젝트 폴더**에 복사
4. `specs/<project>.spec.md` 작성(또는 제공된 가계부 예시 사용)
5. 하네스 실행: `bash harness/run-loop.sh gyebu-app`
6. 성공 시 다음 프로젝트로, 5회 초과 실패 시 로그 확인 후 스펙/프롬프트 보정

상세 프롬프트는 `실행프롬프트.md` 참고.
